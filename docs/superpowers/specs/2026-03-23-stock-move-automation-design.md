# 재고이동 자동화 설계

## 목표

기존 재고이동 자동화 스크립트의 동작을 그대로 유지하면서 로더 모듈 구조로 옮긴다. 메인 페이지 GUI, 검증 단계, 배치 저장, edit 페이지 자동 처리, 리포트, 팝업 무시, iframe 대응까지 포함한다.

## 접근 방식

원본 스크립트를 그대로 붙여 넣는 대신, 테스트 가능한 순수 함수와 실제 브라우저 런타임을 분리하는 방식으로 모듈화한다. 기능 회귀를 막기 위해 입력 파싱, 검증 판정, 목적지 그룹화, 저장 파라미터 생성, 상태 집계를 별도 함수로 분리하고, 페이지 주입과 네트워크 호출은 런타임 계층에서만 수행한다.

## 모듈 구조

- `config/registry.json`
  - `stock-move-automation` 모듈 엔트리를 추가한다.
- `modules/stock-move-automation/meta.json`
  - 표시 이름 `재고이동 자동화`, 엔트리 파일, 의존성 정보를 기록한다.
- `modules/stock-move-automation/main.js`
  - 로더 계약을 따르는 진입점과 순수 헬퍼를 함께 export 한다.
- `tests/stock-move-automation.test.js`
  - 순수 함수와 모듈 계약을 검증한다.
- `tests/loader.test.js`
  - 레지스트리 및 메타 노출을 검증한다.

## 내부 책임 분리

`main.js` 내부는 최소한 다음 책임으로 나눈다.

- 환경/페이지 판별
  - 메인 페이지, edit 페이지, 홈 여부를 안전하게 판별한다.
- 저장소 래퍼
  - localStorage 키, 읽기/쓰기, 초기화, 상태 집계를 한곳에서 관리한다.
- 입력/검증 엔진
  - 탭 구분 입력 파싱, 중복 병합, 검색 결과 매칭, 재고 부족 판정을 순수 함수로 분리한다.
- 배치 실행 엔진
  - 목적지별 그룹화, 출발지별 배치 요청 분리, HTML 파싱 기반 저장 파라미터 생성을 담당한다.
- UI 렌더링
  - 메인 페이지 GUI, 검증 결과, 실행 상태, 완료 리포트를 생성한다.
- 팝업 오버라이드
  - 실행 중 `confirm/alert` 무시와 iframe 재적용 로직을 유지한다.

## 데이터 흐름

1. 사용자가 메인 페이지 GUI에 데이터를 입력한다.
2. 입력 텍스트를 파싱하고 같은 출발지, 상품코드, 목적지 조합은 수량을 합산한다.
3. 출발지 로케이션 단위로 검색 요청을 보내고 결과를 캐시한다.
4. 각 항목을 검색실패, 상품없음, 재고부족, 처리가능 상태로 판정한다.
5. 문제 항목은 리포트용 에러로 남기고, 처리 가능한 항목만 목적지별 큐로 저장한다.
6. 실행 중에는 큐를 소모하면서 직접 저장 API를 호출하고 통계와 로그를 갱신한다.
7. 큐가 비면 실행 상태를 종료하고 완료 리포트를 표시한다.

## 최적화 포인트

- 같은 출발지 로케이션 검색은 한 번만 수행한다.
- 같은 출발지, 상품코드, 목적지 조합은 시작 전에 병합해 요청 건수를 줄인다.
- DOM 업데이트는 상태 전환 시점으로 묶고, 문자열 템플릿 기반 렌더링을 유지해 중복 조작을 줄인다.
- localStorage 키 접근은 래퍼를 통해 통일해 중복 JSON 파싱을 줄인다.
- 배치 저장 전용 파라미터 생성 함수를 분리해 edit 페이지 방식과 직접 API 방식을 구분한다.

## 오류 처리

- 검색 실패, 상품 미존재, 재고 부족은 모두 건너뛰고 에러 로그에 누적한다.
- 배치 저장 실패와 통신 오류도 개별 항목 단위로 에러를 남긴다.
- 목적지 로케이션을 찾지 못한 경우 해당 배치 전체를 실패 처리한다.
- 실행 중지 시 남은 큐는 더 진행하지 않고 사용자가 초기화하거나 다시 시작할 수 있게 둔다.

## 테스트 전략

- 직접 사이트를 조작하는 자동 테스트는 추가하지 않는다.
- 순수 함수와 계약 검증 중심으로 커버한다.
  - 입력 파싱 및 중복 병합
  - 검색 결과 기반 검증 판정
  - 목적지별/출발지별 그룹화
  - 저장 파라미터 직렬화
  - 목적지 선택 매칭
  - 로그/통계 상태 갱신
  - 모듈 export 계약
- 브라우저 실제 동작 확인은 프론트 확인 단계에서 진행한다.

## 배포 원칙

- 구현이 끝나면 `tools/release.ps1`로 버전, changelog, commit, push를 함께 처리한다.
- 관리창 노출은 `main` 피드 기준이므로 최종적으로 `origin/main` 반영이 필요하다.
