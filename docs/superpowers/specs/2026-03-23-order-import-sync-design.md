# 연동데이터 불러오기 설계

## 목표

기존 주문가져오기 자동화 스크립트의 동작을 그대로 유지하면서 로더 모듈 구조로 옮긴다. 신규주문수 `0초과` 판매처만 대상으로 프리뷰를 만들고, 패널 UI, 자동 확인, 결과 파싱, 완료 감지, 세션 재개까지 포함한다.

## 선택한 접근

원본 스크립트를 그대로 감싸는 대신, 테스트 가능한 순수 로직과 브라우저 런타임을 분리하는 방식으로 모듈화한다. 대상 스캔, 정렬, 결과 테이블 파싱, 성공/실패 집계, 완료 판정, 저장소 상태 전이는 순수 함수로 분리하고, `confirm/alert/prompt` 패치와 DOM 주입, 폴링, iframe 패치는 런타임 계층에서 담당한다.

## 모듈 구조

- `config/registry.json`
  - `order-import-sync` 모듈 엔트리를 추가한다.
- `modules/order-import-sync/meta.json`
  - 표시 이름 `연동데이터 불러오기`, 엔트리 파일, 버전 정보를 기록한다.
- `modules/order-import-sync/main.js`
  - 로더 진입점, 순수 헬퍼, 브라우저 런타임을 함께 export 한다.
- `tests/order-import-sync.test.js`
  - 스캔, 정렬, 결과 파싱, 완료 판정, 상태 전이, 모듈 계약을 검증한다.
- `tests/loader.test.js`
  - 레지스트리와 메타 노출을 검증한다.

## 내부 책임 분리

`main.js` 내부는 다음 역할로 나눈다.

- 환경/페이지 판별
  - `site230main.jsp`에서만 실행하고, 다른 화면에서는 즉시 종료한다.
- 저장소 래퍼
  - localStorage 키, 큐, 진행 인덱스, 현재 판매처, 결과 맵, 자동확인 상태를 한곳에서 관리한다.
- 스캔/프리뷰 엔진
  - DOM에서 판매처 코드, 이름, 신규주문수를 읽고 `0초과` 대상만 정렬해 프리뷰 데이터로 만든다.
- 결과 감지 엔진
  - 현재 문서와 동일 도메인 iframe 안에서 결과 테이블과 메시지를 찾고, 성공/실패 집계를 계산한다.
- 완료 판정 엔진
  - 결과 테이블 감지와 신규주문수 감소를 함께 사용해 완료를 판정한다.
- UI 렌더링
  - 패널 생성, 축소/확장, 상태 텍스트, 로그, 프리뷰 상태 갱신을 담당한다.
- 자동확인/iframe 패치
  - `confirm/alert/prompt` 자동 처리를 유지하고 iframe 로드 이후에도 재적용한다.

## 데이터 흐름

1. 사용자가 패널에서 스캔을 실행하거나 진입 시 자동 프리뷰가 만들어진다.
2. DOM에서 판매처 행을 읽어 신규주문수 `0초과`이면서 버튼이 있는 판매처만 큐로 저장한다.
3. 사용자가 시작하면 큐 인덱스를 0으로 초기화하고 자동 처리 상태를 켠다.
4. 각 판매처에 대해 주문가져오기 버튼을 클릭하고, 자동 확인 패치가 뜨는 대화상자를 처리한다.
5. 결과 테이블 또는 완료 메시지를 현재 문서와 iframe에서 찾는다.
6. 결과가 없더라도 신규주문수가 줄어들면 백업 완료로 판단한다.
7. 판매처별 결과를 저장하고 프리뷰 상태를 갱신한 뒤 다음 판매처로 진행한다.
8. 큐가 끝나면 전체 성공/실패 요약을 표시하고 자동 상태를 종료한다.

## 최적화 포인트

- 프리뷰 데이터는 스캔 시점에 정렬과 필터링을 끝내고 큐에 저장해 반복 계산을 줄인다.
- 결과 감지는 문서와 iframe 모두에 대해 동일한 파서 함수를 재사용한다.
- 로딩 감지, 결과 감지, 주문수 감소 감지를 분리해 폴링 루프를 단순화한다.
- 저장소 키 접근을 래퍼로 통일해 중복 JSON 파싱과 상태 갱신 분산을 줄인다.
- UI 갱신은 상태, 로그, 프리뷰 셀 업데이트로 분리해 각 변경 범위를 최소화한다.

## 오류 처리

- 판매처 버튼을 찾지 못하면 해당 판매처를 스킵으로 기록하고 다음으로 진행한다.
- 결과 테이블을 찾지 못해도 주문수 감소가 감지되면 성공으로 처리한다.
- 결과 대기 시간 초과 시 타임아웃 상태로 기록하고 다음 판매처로 진행한다.
- 사용자가 정지를 누르면 활성 상태와 처리 상태를 즉시 내리고 현재 판매처를 비운다.
- iframe 접근이 불가능한 경우는 무시하고 현재 문서 기준 감지만 유지한다.

## 테스트 전략

- 실제 사이트 조작 자동화는 추가하지 않는다.
- 순수 함수와 계약 검증 중심으로 커버한다.
  - DOM 스캔 결과를 프리뷰 큐 형태로 정리하는 함수
  - 판매처 정렬 및 필터링
  - 결과 행 파싱 및 성공/실패 집계
  - 완료 메시지 파싱
  - 주문수 감소 기반 완료 판정
  - localStorage 상태 전이
  - 모듈 export 계약
- 브라우저 실제 동작 확인은 사용자가 프론트에서 테스트한다.

## 배포 원칙

- 구현이 끝나면 `tools/release.ps1`로 버전, changelog, commit, push를 함께 처리한다.
- 사용자가 로더에서 바로 테스트할 수 있도록 기본적으로 같은 작업 안에서 `main`까지 반영한다.
